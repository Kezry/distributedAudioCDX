# distributedAudioCDX 开发计划

## 1. 项目目标

设计并实现一个局域网分布式音频系统：

- Windows 端作为系统音频输出源。
- 多个 Android 4.4+ 设备作为声音端。
- 支持普通低延迟声卡模式。
- 支持 DLNA 单机和多机模式。
- 支持 2.0、2.1、5.1、7.1 等声道布局。
- 多设备播放时尽量保持毫秒级同步，避免左右声道延迟不一致。

## 3. 阶段计划

### 阶段 0：仓库与设计准备

- 建立三个顶级模块目录。
- 明确模块职责。
- 明确协议边界。
- 明确 MVP 和正式版差异。
- 产出架构文档和开发路线图。

交付物：

- 根目录 README。
- 模块 README。
- 架构计划文档。
- 初版协议文档。

### 阶段 1：MVP 声卡模式验证

目标：先证明 Windows 可以把音频稳定、低延迟地推送到 Android 声音端。

Windows 发送端：

- 实现设备扫描。
- 实现 UDP PCM 音频发送。
- 先用 WASAPI Loopback 或临时音频源做验证。
- 支持选择设备和左右声道映射。

Android 声音端：

- 启动后广播设备信息。
- 接收 UDP PCM 音频包。
- 使用 AudioTrack 播放。
- 支持左/右声道提取。
- 提供简单状态页面或日志。

验证指标：

- 单设备可连续播放 30 分钟。
- 局域网内默认延迟目标 40ms - 100ms。
- 丢包或乱序时不中断播放。

### 阶段 2：多设备同步

目标：支持多个 Android 声音端分别承担不同声道，并尽量同步播放。

工作内容：

- 音频包加入 sequence、stream timestamp、play_at 时间。
- Windows 端实现主时钟。
- Android 端实现本地时钟偏移估算。
- 实现 jitter buffer。
- 支持每台设备手动/自动延迟补偿。
- 实现轻量重采样或播放速率微调，修正时钟漂移。

验证指标：

- 左右声道同步误差目标小于 5ms。
- 稳定网络下目标小于 2ms。
- 长时间播放无明显漂移。

### 阶段 3：高级声道布局

目标：支持 2.1、5.1、7.1 和自定义声道矩阵。

工作内容：

- 定义标准声道枚举。
- Windows UI 支持高级模式。
- 支持每个设备绑定一个或多个声道。
- 支持声道下混、复制、静音和音量校准。
- 支持低音声道处理策略。

验证指标：

- 设备声道映射清晰可见。
- 切换声道布局时不中断或最小化中断。
- 每个声道可单独测试。

### 阶段 4：DLNA 模式

目标：让 Android 声音端可以作为 DLNA Renderer 使用，并支持增强多机模式。

工作内容：

- Android 声音端实现 UPnP/DLNA 设备发现与控制。
- 支持标准 DLNA 单机播放。
- Windows 发送端扫描 DLNA 设备。
- 多机 DLNA 模式通过自定义同步协议补强。

注意：

标准 DLNA 不保证多设备严格同步。多设备左右声道播放不能只依赖标准 DLNA，需要使用自定义同步层。

### 阶段 5：Windows 虚拟声卡正式化

目标：让 Windows 发送端成为真正的系统音频输出设备。

工作内容：

- 调研并确定虚拟声卡驱动路线。
- 实现或集成 Windows 虚拟音频驱动。
- 用户态服务接收驱动音频流。
- 支持 44.1kHz/48kHz、16-bit/24-bit、2.0/5.1/7.1。
- 完成驱动签名、安装、卸载和兼容性测试。

建议路线：

- 原型期：WASAPI Loopback 或第三方虚拟声卡。
- 产品期：自研或定制 Windows 虚拟音频驱动。

## 4. 关键风险

- Windows 虚拟声卡驱动开发和签名成本高。
- Android 4.4 不同设备音频延迟差异大。
- WiFi 网络抖动会直接影响低延迟和同步。
- 标准 DLNA 不适合严格多设备同步。
- 多设备同步需要持续处理时钟漂移。

4. 实现 Windows 发送端最小扫描和推流。
5. 用两台 Android 设备验证左右声道同步。
